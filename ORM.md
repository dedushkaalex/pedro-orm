Как работают ORM: путь от кода до БД на примере Drizzle

ORM (Object-Relational Mapper) — это прослойка, которая превращает вызовы на языке программирования в
SQL-запросы и обратно — строки из БД в объекты языка. Drizzle — это "тонкий" ORM, ближе к типизированному
query builder'у, чем к классическим Active Record / Data Mapper (Prisma, TypeORM, Sequelize). Разберём
по слоям.

1. Определение схемы (Schema Definition)

import { pgTable, serial, text, integer } from 'drizzle-orm/pg-core';

export const users = pgTable('users', {
id: serial('id').primaryKey(),
name: text('name').notNull(),
age: integer('age'),
});

Что здесь происходит:

- pgTable создаёт runtime-объект, который описывает таблицу: имя, колонки, типы, ограничения.
- Одновременно через TypeScript-магию (InferSelectModel<typeof users>, InferInsertModel) выводятся
  статические типы строк.
- В отличие от Prisma, нет кодогенерации — нет отдельного шага prisma generate. Схема — это обычный
  TS-файл, а типы выводятся компилятором.

2. Построение запроса (Query Builder)

const result = await db
.select({ id: users.id, name: users.name })
.from(users)
.where(eq(users.age, 25))
.limit(10);

На этом этапе вы не строите строку SQL. Вы строите AST (Abstract Syntax Tree) — дерево объектов,
описывающее запрос:

- select(...) создаёт объект PgSelectBuilder.
- .from(users) добавляет узел FROM.
- .where(eq(...)) — узел WHERE с условием (eq возвращает объект SQL с операндами).
- .limit(10) — узел LIMIT.

Цепочка возвращает объект-билдер. Каждый метод иммутабельно/мутабельно дополняет внутреннее состояние.
Запрос ещё не выполнен — он "ленивый" (thenable).

3. Превращение AST → SQL (Dialect / Compiler)

Когда вы делаете await (или .execute()), срабатывает thenable, и билдер передаёт AST диалекту:

PgDialect.sqlToQuery(ast) → { sql: string, params: unknown[] }

Диалект знает специфику конкретной СУБД:

- PostgreSQL использует $1, $2, $3 для плейсхолдеров.
- MySQL использует ?.
- SQLite — ? или именованные.
- Различия в синтаксисе: RETURNING, ON CONFLICT, кавычки идентификаторов.

Результат — параметризованный SQL + массив значений:
select "id", "name" from "users" where "age" = $1 limit $2
-- params: [25, 10]

Параметризация критична: она защищает от SQL-инъекций (значения не подставляются строкой, а уходят
отдельным каналом в драйвер).

4. Драйвер БД (Database Driver)

Drizzle сам не общается с БД напрямую. Он передаёт { sql, params } в драйвер, который вы подключили:

import { drizzle } from 'drizzle-orm/node-postgres';
import { Pool } from 'pg';

const db = drizzle(new Pool({ connectionString: '...' }));

Драйверы бывают разные:

- pg (node-postgres) — TCP, классика.
- postgres (postgres.js) — TCP, быстрее.
- @neondatabase/serverless — HTTP/WebSocket для serverless.
- better-sqlite3 — синхронный нативный binding.
- @libsql/client — HTTP/WebSocket для Turso.

Драйвер берёт на себя:

- Пул соединений.
- Сериализацию параметров в wire-формат конкретной БД.
- Сетевой обмен по протоколу СУБД (PostgreSQL Frontend/Backend Protocol, MySQL Protocol и т.д.).

5. Выполнение в СУБД

На стороне базы:

1. Парсер разбирает SQL в свой внутренний AST.
2. Планировщик строит план выполнения (использовать индекс, делать seq scan, какой join алгоритм и т.д.).
3. Executor исполняет план — читает страницы с диска / из кэша, фильтрует, сортирует, формирует
   результат.
4. Возвращает rows в wire-формате.

Drizzle на этот этап не влияет вообще — это уже территория СУБД.

6. Маппинг результата обратно (Result Mapping)

Драйвер получает байты, парсит их в массив объектов с примитивами (string, number, Buffer...). Drizzle
получает "сырой" результат и:

- Сопоставляет имена колонок в SQL с ключами, которые вы указали в select({...}).
- Применяет колоночные трансформации: например, для timestamp создаёт Date, для jsonb — парсит JSON, для
  пользовательских customType — вызывает ваш fromDriver.
- Если использовалась db.query.users.findMany({ with: { posts: true } }) (relational API), Drizzle ещё и
  склеивает плоский результат join'а в иерархию объектов.

На выходе — типизированный массив, который точно совпадает с тем, что обещал TypeScript на этапе
билдинга.

Сводная диаграмма

[ TS-код ]
│ db.select().from().where()
▼
[ Query Builder ] ── строит AST (объекты SQL/Column/Table)
│
▼
[ Dialect / Compiler ] ── AST → { sql, params } для конкретной СУБД
│
▼
[ Driver (pg / postgres / libsql) ] ── сетевой протокол + пул
│
▼
[ Database ] ── parse → plan → execute → rows
│
▼
[ Driver ] ── wire-формат → JS-объекты
│
▼
[ Drizzle Mapper ] ── трансформации типов + relational shaping
│
▼
[ Типизированный результат ]

Чем Drizzle отличается от "тяжёлых" ORM

┌───────────────────────────┬────────────────────────────────┬───────────────────────────────────────┐
│ Аспект │ Drizzle │ Prisma / TypeORM │
├───────────────────────────┼────────────────────────────────┼───────────────────────────────────────┤
│ Схема │ TS-объекты, типы через │ Отдельный DSL + кодогенерация │
│ │ inference │ │
├───────────────────────────┼────────────────────────────────┼───────────────────────────────────────┤
│ Запрос │ Query builder, близок к SQL │ Декларативный объект { where, include │
│ │ │ } │
├───────────────────────────┼────────────────────────────────┼───────────────────────────────────────┤
│ Runtime │ Тонкий слой над драйвером │ Свой engine (у Prisma — Rust-бинарь) │
├───────────────────────────┼────────────────────────────────┼───────────────────────────────────────┤
│ Identity map / unit of │ Нет │ Есть (TypeORM), частично (Prisma) │
│ work │ │ │
├───────────────────────────┼────────────────────────────────┼───────────────────────────────────────┤
│ Миграции │ drizzle-kit (отдельно) │ Встроено │
└───────────────────────────┴────────────────────────────────┴───────────────────────────────────────┘

Drizzle сознательно не реализует Identity Map, Unit of Work, lazy-loading сущностей — это убирает магию и
делает поведение предсказуемым ценой того, что часть работы (например, кэширование, отслеживание
изменений) ложится на вас.

Ключевая идея

ORM — это последовательность трансформаций: типизированный API → AST → SQL+params → wire-протокол →
результат БД → обратный маппинг. Drizzle делает каждый шаг тонким и прозрачным, поэтому его часто
называют не ORM, а "headless типизированный SQL".
