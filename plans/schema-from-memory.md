# Schema — план самостоятельной сборки по памяти

Документ-памятка для повторной реализации модуля `src/schema/` без подсказок
кода. Никаких сигнатур, реализаций или псевдокода — только цели, инварианты
и порядок шагов. Если застрял — открой нужный шаг и проверь, что задача
поставлена корректно. Если что-то «странно ломается» — открой раздел
«Грабли».

---

## 1. Что мы строим

**Type-safe DSL для описания таблиц** + набор type-level «инференсеров»,
вычисляющих TS-типы из этого описания.

Schema описывается один раз. Из неё автоматически выводятся:

- **тип строки таблицы** — что вернёт `SELECT * FROM ...`
- **тип входа для INSERT** — что нужно передать в `INSERT INTO ...`

Никакой runtime-логики ORM в этом слое нет — фабрики собирают plain-объекты,
вся «магия» живёт на уровне типов.

---

## 2. Критерии готовности

Сценарий должен компилироваться и давать ожидаемые типы:

- Можно описать таблицу с колонками (int, text, real, boolean, blob).
- Колонку можно пометить как `nullable`.
- Колонку можно пометить как `primaryKey`.
- Из дескриптора таблицы выводится **row-тип**: все колонки обязательны;
  nullable отражается как `T | null`.
- Из дескриптора таблицы выводится **insert-тип**: PK опционален, остальное
  обязательно.

Должны давать ошибку компиляции:

- [ ] Опечатка в имени SQL-типа (например, `"intger"`).
- [ ] Применение `nullable` к уже-nullable колонке.
- [ ] Применение `primaryKey` к уже-PK колонке.
- [ ] Комбинация nullable + primaryKey в любом порядке (PK не может быть
      nullable).
- [ ] Использование row-типа там, где ожидается дескриптор таблицы (и наоборот).

Должны выводиться корректно:

- [ ] Имя таблицы как литерал (`"users"`, не `string`).
- [ ] Колонки в карте — каждая с конкретными литеральными параметрами.
- [ ] `Record<string, ColumnDef>` без аргументов компилируется и принимает
      любые комбинации nullable/PK.

## 3. План по шагам

### Шаг 1. Алфавит SQL-типов

**Цель:** ограничить разрешённые имена типов колонок узким union-ом
строковых литералов.

**Что помнить:** взять минимум — то, что нативно есть у SQLite (5 типов).
Это будет «единственный источник правды» для имён в схеме.

---

### Шаг 2. Дескриптор колонки

**Цель:** интерфейс для описания одной колонки. Должен трекать **на уровне
типов** три параметра: SQL-тип, nullable-флаг, PK-флаг.

**Что помнить:**

- Три дженерика: один на тип, два на boolean-флаги.
- Для каждого дженерика — соответствующее **runtime-поле** с тем же литералом.
  Это нужно, чтобы потом conditional types (`extends true`) могли по
  значению поля различить случаи.
- Дополнительное поле под default-значение (опциональное, на будущее) —
  пока не используется, но зарезервируй.
- **Дефолты дженериков — `boolean`, не `false`.** См. «Грабли», п. 1.

---

### Шаг 3. Фабрики колонок

**Цель:** одна функция на каждый SQL-тип. Возвращает дескриптор с
дефолтными флагами (не nullable, не PK).

**Что помнить:**

- Возвращаемый тип — **узкий литерал** для всех трёх параметров. Это
  критично: голый `ColumnDef` обрушит весь дальнейший inference.
- Имя функции может совпадать с примитивом TS — это не конфликтует, типы
  и значения в разных namespace.

---

### Шаг 4. Модификаторы

**Цель:** две функции — переключают флаг nullable / PK у переданной колонки.

**Что помнить:**

- Каждый модификатор должен **отказывать**, если флаг уже `true`. Это
  свойство выражается через тип входного параметра.
- Соседний флаг (для nullable — это PK, и наоборот) надо пробросить через
  дженерик, чтобы он не сбросился.
- Реализация **иммутабельная**: создаём новый объект, не мутируем входной.

---

### Шаг 5. Дескриптор таблицы

**Цель:** интерфейс «имя таблицы + карта колонок».

**Что помнить:**

- Два дженерика: имя (узкий string-литерал) и карта колонок.
- Карта колонок ограничена `Record<string, ColumnDef>`. Этот constraint
  работает только если у `ColumnDef` дефолты широкие — иначе на этом шаге
  отвалятся все nullable/PK колонки.

---

### Шаг 6. Фабрика таблицы

**Цель:** функция от имени и карты колонок, возвращающая дескриптор таблицы.

**Что помнить:**

- Дженерики обязательны — иначе TS «округлит» имя до `string`, а карту до
  `Record<string, ColumnDef>` без литералов.
- Никаких `as Type` — только дженерики делают сохранение литералов корректно.

---

### Шаг 7. Маппинг SQL → TS

**Цель:** объект-словарь, у которого ключи — имена SQL-типов из шага 1,
значения — соответствующие TS-типы.

**Что помнить:**

- integer и real → числа.
- text → строка.
- boolean → boolean (преобразование в 0/1 для хранения — задача driver-а,
  не схемы).
- blob → бинарный буфер.
- TS должен ругаться, если в словаре пропущен какой-то ключ из алфавита
  (это и есть роль ограничения через type union в шаге 1).

---

### Шаг 8. Инференс одной колонки

**Цель:** type alias, который из дескриптора одной колонки вычисляет её
TS-тип, учитывая nullable.

**Что помнить:**

- Conditional type: проверяем флаг nullable; если он `true` — объединяем
  базовый TS-тип с `null`.
- Базовый TS-тип берётся индексным доступом к словарю из шага 7 по полю
  типа дескриптора.

---

### Шаг 9. Инференс строки

**Цель:** type alias, который из дескриптора таблицы выводит тип строки —
все колонки обязательны.

**Что помнить:**

- Mapped type по ключам карты колонок.
- Каждое значение прогоняется через инференсер из шага 8.

---

### Шаг 10. Инференс input-а для INSERT

**Цель:** type alias, который выводит тип строки для INSERT — PK становится
опциональным, остальное обязательно.

**Что помнить:**

- Раздели колонки на две группы: PK и не-PK.
- Для разделения используй **key remapping в mapped type** (`as ... ? K
: never`). `never` в позиции `as` выбрасывает ключ из результирующего
  типа.
- Не-PK ключи — обязательные.
- PK ключи — опциональные (модификатор `?` в mapped type).
- Склей две группы пересечением (`&`).

## 4. Грабли (открывать при странных ошибках)

### 1. `Type 'true' is not assignable to type 'false'`

Симптом: TS ругается, что `true` не лезет в `false`, обычно при
использовании `Record<string, ColumnDef>` или `table(...)` с PK/nullable
колонками.

Причина: дефолты дженериков `ColumnDef` слишком узкие (`= false`). При
использовании имени без параметров TS подставляет именно `false`, и тогда
колонка с `_pk: true` или `_nullable: true` не проходит constraint.

Лечение: дефолты дженериков должны быть **`boolean`** (= constraint), а не
конкретный литерал. Узкие литералы остаются в фабриках, где они нужны для
последующего инференса.

Правило большого пальца: **дефолт generic-параметра — это буквальная
подстановка, а не «гибкая заглушка». Дефолт должен быть равен constraint,
а не самому частому конкретному значению.**

---

### 2. `InferRow` выдаёт `any` или `unknown`

Симптом: ожидаешь `{ id: number; ... }`, получаешь `Record<string, any>`
или просто `any`.

Причина: где-то по дороге потерялись литеральные типы. Чаще всего — явная
аннотация `: TableDef<string, any>` или `: ColumnDef` без параметров.
Аннотация типа «расширяет» конкретный объект до общего интерфейса.

Лечение: убрать аннотацию и положиться на дженерики фабрик, либо
использовать `satisfies` вместо `:` (он проверяет соответствие, но
сохраняет узкий выведенный тип).

---

### 3. Модификатор «принимает» уже-применённую колонку

Симптом: `nullable(nullable(text()))` или `primaryKey(primaryKey(...))`
компилируется, хотя не должно.

Причина: в сигнатуре модификатора входной параметр не ограничен по
соответствующему флагу.

Лечение: входной тип должен явно требовать `false` для своего флага.

---

### 4. `nullable(primaryKey(...))` компилируется, а должна нет

Симптом: TS пропускает заведомо невалидную комбинацию.

Причина: один из модификаторов не пробрасывает соседний флаг через
дженерик и принимает «любой» вместо требования `false`.

Лечение: оба модификатора должны жёстко требовать `false` для своего
собственного флага. Тогда после первого вызова второй модификатор не
сможет применить себя — и порядок не имеет значения, обе перестановки
будут отвергнуты.

---

### 5. `as` в mapped type не делает то, что ожидаешь

Это **key remapping** (с TS 4.1), а **не** type assertion. В выражении
`[K in keyof T as <NewKey>]: ...`:

- если `<NewKey>` — другой ключ, происходит переименование;
- если `<NewKey>` — `never`, ключ **выбрасывается** из результирующего типа;
- если `<NewKey>` — template literal, ключ переписывается по шаблону.

Это ровно тот механизм, через который `InferInsert` фильтрует PK / не-PK
колонки в две разные половины.

## 5. Ссылки на документацию TS

По механике сохранения литеральных типов и type widening — это
ключевая концепция, на которой построен весь модуль.

- **TypeScript Handbook — Literal Types**
  https://www.typescriptlang.org/docs/handbook/2/everyday-types.html#literal-types
  — что такое `"hello"` / `42` / `true` как типы и где они сохраняются.

- **TypeScript Handbook — Generics (Generic Constraints)**
  https://www.typescriptlang.org/docs/handbook/2/generics.html
  — почему `<N extends string>` ведёт себя иначе, чем `string` в
  позиции параметра.

- **TypeScript 2.1 Release Notes — Better inference for literal types**
  https://www.typescriptlang.org/docs/handbook/release-notes/typescript-2-1.html
  — первоисточник механизма. Здесь введён термин **type widening**.

- **TypeScript 5.0 Release Notes — Const Type Parameters**
  https://www.typescriptlang.org/docs/handbook/release-notes/typescript-5-0.html#const-type-parameters
  — `<const T extends ...>` для нерасширяющего вывода объектов и массивов.

- **TypeScript Wiki — What is type widening?**
  https://github.com/microsoft/TypeScript/wiki/FAQ#what-is-type-widening
  — короткий разбор механики на примерах.

Краткая выжимка: расширение (widening) — это когда `"users"`
автоматически превращается в `string`. Происходит в «mutable» позициях
(параметры функций без дженериков, `let`-переменные, обычные поля
объектов). **Не** происходит для `const`-переменных, `readonly`-полей,
аргументов в позиции дженерика и аргументов под `<const T>` (TS 5.0+).

## 6. Порядок чтения, если забыл, на каком шаге остановился

1. Открой `src/schema/columns.ts` — там должно быть: алфавит, интерфейс
   колонки, фабрики, два модификатора. Если что-то из этого отсутствует —
   шаги 1–4.
2. Открой `src/schema/table.ts` — там дескриптор и фабрика таблицы. Если
   нет — шаги 5–6.
3. Открой `src/schema/infer.ts` — там словарь маппинга и три type alias-а.
   Если нет — шаги 7–10.

Каждый шаг **независимо проверяется** на сценарии из раздела «Критерии
готовности». Если сценарий после шага N не работает — баг в шагах ≤ N.
