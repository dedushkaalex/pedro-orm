# Архитектурный обзор — High-Level Design

## Назначение

Документ объясняет, из каких компонентов собран `pedro-orm`, кто за
что отвечает и как они общаются через Effect-TS-примитивы. Дальше по
тексту встречаются короткие учебные блоки (**📘 учебник**) с
объяснением используемых концепций — Effect-TS и реляционных БД.

## Зачем вообще ORM как слой

Сырой драйвер БД (PGlite, better-sqlite3) умеет одну операцию: «выполни
SQL, верни строки». Этого мало для прикладного кода:

- разные движки используют **разный синтаксис плейсхолдеров** (`$1` vs
  `?`), кавычки идентификаторов, типы колонок;
- разные движки **по-разному кодируют значения** (например, у SQLite
  нет `BOOLEAN`, его заменяют `INTEGER 0/1`);
- ошибки приходят в виде нетипизированных `unknown` с разными кодами
  (`23505` у Postgres, `SQLITE_CONSTRAINT_UNIQUE` у SQLite);
- нужно где-то аккуратно открывать и закрывать соединение, чтобы при
  любой ошибке оно не текло.

ORM-слой решает эти задачи **до** того, как сырые данные попадут в
бизнес-логику. Сама бизнес-логика остаётся переносимой между движками.

## Компоненты

### Driver

**Ответственность**: единый контракт на «выполнить SQL и получить
результат».

**Вход**: SQL-строка и массив параметров.

**Выход**: `RawResult` (массив строк, число затронутых строк, опционально
ID последней вставки).

**Владеет**: контрактом запроса и набором тегов-ошибок, которые
гарантированно возвращаются (а не вылетают как `throw`).

Driver — **зависимость**, а не реализация. Бизнес-код просит «дайте мне
Driver», а кто его предоставит (PGlite, SQLite, мок в тестах) —
решается на этапе сборки приложения.

> 📘 **учебник: Context.Tag (Effect-TS)**.
> Чтобы передавать зависимости (логгер, БД, конфиг) без явного
> протаскивания через параметры, в Effect-TS заводят «тег» — уникальный
> токен, по которому в рантайме ищется реализация. Любой `Effect`
> декларирует свои зависимости в типе. Это форма DI, где компилятор
> следит за тем, чтобы при запуске программы все теги были закрыты.

### Dialect

**Ответственность**: знать различия в **SQL-синтаксисе** конкретного
движка.

**Вход**: запросы от Driver и схемы (например, «как процитировать
идентификатор», «как сгенерировать плейсхолдер #3»).

**Выход**: строки-фрагменты SQL, корректные для конкретного движка.

**Владеет**: булевыми флагами поддержки фич (например,
`supportsReturning`) и маппингом SQL-типов в их движковое имя.

### Codec

**Ответственность**: знать различия в **кодировании значений** между
движками — на уровне TS-значений, а не SQL-текста.

**Вход**: TS-значение (например, `boolean`, `Date`, объект JSON).

**Выход**: значение, которое уже можно передать в драйвер как параметр
запроса. И обратное преобразование при чтении.

**Владеет**: парами `encode/decode` для конкретного SQL-типа и
конкретного диалекта.

> 📘 **учебник: чем codec отличается от dialect**.
> Dialect работает с **SQL-текстом** (`"$1"`, `"BIGSERIAL"`,
> `"\"users\""`). Codec работает с **значениями параметров**
> (`true → 1`). Это разные оси различий между движками, и они
> разнесены, чтобы их можно было комбинировать независимо.

### Errors

**Ответственность**: преобразовать произвольный `unknown`-кейс ошибки
драйвера в один из заранее объявленных tagged-классов.

**Владеет**: иерархией ошибок (`DbError`, `UniqueViolationError`,
`ForeignKeyViolationError`, `StatementTimeoutError`, `CodecError`,
`NotFoundError`, `TooManyError`) и сводным типом `DriverError`.

> 📘 **учебник: Data.TaggedError**.
> В Effect-TS ошибки — это значения в типе `Effect<A, E, R>`. Чтобы
> различать кейсы, заводят классы-теги: у каждого есть поле `_tag` со
> своим литералом. Тогда `Effect.catchTag("UniqueViolationError", …)`
> ловит конкретный случай, оставляя остальные распространяться. Это
> аналог дискриминированных union-ов в чистом TS, но интегрированный с
> рантаймом Effect.

### Concrete Drivers (`PGlite`, `SQLite`)

**Ответственность**: на основе конкретной библиотеки построить
реализацию `DriverImpl`. Открыть соединение, аккуратно закрыть его,
маппить ошибки конкретного движка в общий `DriverError`.

**Владеют**: жизненным циклом подключения и переводом
ошибок-конкретного-движка в доменные.

### Tracing Layer

**Ответственность**: middleware, который оборачивает уже существующий
`Driver` и добавляет span'ы OpenTelemetry на каждый запрос. Не меняет
поведение — только добавляет наблюдаемость.

**Владеет**: правилом «как именно называется span и какие атрибуты на
него навешиваются».

### Schema

**Ответственность**: декларативно описать таблицы и колонки на уровне
типов и значений. Из этого описания type-level выводятся форма `Row` и
форма `Insert`.

Подробности — в `docs/lld/schema.md`.

## Диаграмма компонентов

```
                бизнес-код (program в lesson-1.ts)
                              │
                              │ yield* DriverDependency
                              ▼
                ┌──────────────────────────────┐
                │   DriverDependency (Tag)     │
                └──────────────┬───────────────┘
                               │
        ┌──────────────────────┴──────────────────────┐
        │            TracingLayer (опционально)        │
        │     добавляет span "db.query" к каждому      │
        │     вызову executeRaw, не меняя контракт     │
        └──────────────────────┬──────────────────────┘
                               │
        ┌──────────────────────┴──────────────────────┐
        │      Concrete driver layer (PGlite | SQLite) │
        │  открывает соединение через acquireRelease,  │
        │  возвращает DriverImpl с executeRaw          │
        └─────────┬────────────────────┬──────────────┘
                  │                    │
                  ▼                    ▼
              ┌────────┐          ┌────────┐
              │ PGlite │          │ SQLite │
              └────────┘          └────────┘
                  │                    │
                  └────────┬───────────┘
                           ▼
                  ┌──────────────────┐
                  │     Dialect      │  ← плейсхолдеры, идентификаторы
                  │     Codec        │  ← кодирование значений
                  │     Errors       │  ← маппинг к DriverError
                  └──────────────────┘
```

## Поток: выполнение запроса

Рассмотрим запрос из `playground/lesson-1.ts`:
`INSERT INTO "users" … VALUES ($1, $2, $3) RETURNING *`.

1. **Бизнес-код** запрашивает у Effect-рантайма `DriverDependency`. В
   контексте уже лежит реализация — её туда положил `Layer`.
2. **Бизнес-код** строит SQL: спрашивает у `dialect.placeholder(n)`,
   как обозначить параметр (`$1` для PG, `?` для SQLite). Сами
   значения параметров приходит как обычный массив.
3. **Tracing Layer** перехватывает вызов `executeRaw`, открывает span
   `db.query`, прокидывает атрибуты (`db.system`, `db.statement`,
   `db.params.count`).
4. **Concrete driver** делает реальный вызов к библиотеке. На PG —
   `pg.query(sql, params)`. На SQLite — `db.prepare(sql).all(...)` или
   `.run(...)` в зависимости от того, читающий это запрос или нет.
5. Если библиотека выбросила исключение, **mapper ошибок** конкретного
   драйвера превращает его в один из `DriverError`. Никаких `throw` —
   вернётся `Effect.fail(...)`.
6. На успехе **concrete driver** оборачивает результат в `RawResult` с
   единообразной формой (`rows`, `affectedRows`, опционально
   `lastInsertRowId`).
7. **Tracing Layer** на выходе пишет в span количество возвращённых и
   затронутых строк.
8. **Бизнес-код** получает `RawResult` и может опционально перехватить
   конкретные ошибки через `Effect.catchTag("UniqueViolationError", …)`.

## Поток: жизненный цикл соединения

```
Программа           Layer.scoped       Driver (PGlite/SQLite)
   │                     │                       │
   │  Effect.scoped      │                       │
   │ ───────────────────►│                       │
   │                     │  acquireRelease       │
   │                     │ ─────────────────────►│
   │                     │       open connection │
   │                     │ ◄─────────────────────│
   │  Driver доступен    │                       │
   │ ◄───────────────────│                       │
   │  ... запросы ...    │                       │
   │                     │                       │
   │  scope finalizes    │                       │
   │ ───────────────────►│                       │
   │                     │  close connection     │
   │                     │ ─────────────────────►│
   │                     │            (всегда!)  │
```

> 📘 **учебник: Scope и acquireRelease**.
> `Scope` — это «область, к концу которой нужно прибраться». Когда
> ресурс взят через `Effect.acquireRelease(open, close)`, Effect
> гарантирует вызов `close` при выходе из области, даже если внутри
> произошла ошибка или прерывание. `Layer.scoped` — это «слой,
> привязанный к Scope», именно так в проекте оформлены оба конкретных
> драйвера: соединение открывается на создание слоя и закрывается,
> когда `Effect.scoped(program)` отрабатывает до конца.

## Контракт между слоями

| Граница               | Что передаётся                                  | Кто формирует                     |
| --------------------- | ----------------------------------------------- | --------------------------------- |
| Бизнес-код → Driver   | `(sql: string, params: ReadonlyArray<unknown>)` | бизнес-код, опираясь на `dialect` |
| Driver → Бизнес-код   | `RawResult` или один из `DriverError`           | concrete driver                   |
| Tracing → Concrete    | прозрачный pass-through `executeRaw`            | tracing layer не меняет аргументы |
| Concrete → библиотека | специфичный API (`pg.query`, `stmt.all`)        | concrete driver                   |
| Библиотека → Concrete | `unknown` ошибка / `unknown` строки             | concrete driver маппит к домену   |

## Композиция слоёв

`Layer` — это «рецепт сборки» зависимости. Слои композируются:

```
TracingLayer.pipe(Layer.provide(SqliteDriver.layer({ path: ":memory:" })))
```

Читать справа налево по стрелке зависимости:

- `SqliteDriver.layer(...)` — реализация `DriverDependency` через
  better-sqlite3
- `TracingLayer` — тоже реализация `DriverDependency`, но требующая,
  чтобы кто-то предоставил «внутренний» `DriverDependency` для
  оборачивания
- `Layer.provide` склеивает: внешний `TracingLayer` получает внутренний
  `SqliteDriver` как свою зависимость

Бизнес-код этого не знает: он просто видит, что
`DriverDependency` доступен. Замена SQLite на PGlite — это замена одной
строки в сборке.

> 📘 **учебник: middleware через Layer**.
> Идея «обернуть существующую реализацию» через слой, который сам
> требует ту же зависимость, что и отдаёт, — стандартный приём в
> Effect-TS. Так же делают логирование, ретраи, кэш, метрики.

## Cross-cutting concerns

- **Ошибки**. Все возможные сбои собраны в `DriverError` (union
  tagged-классов). Бизнес-код может обработать конкретный кейс через
  `catchTag` или дать ошибке распространиться.
- **Трассировка**. Подключается опциональным `TracingLayer`. Бизнес-код
  про него не знает.
- **Логирование**. В примере `lesson-1.ts` используется
  `Effect.annotateLogs({ engine: "sqlite" })` — это аннотация, которая
  будет добавлена ко всем логам в области. Не относится к ORM-слою как
  таковому, но показывает, как поверх него удобно навешивать контекст.

## Ограничения текущей реализации

- Нет query-builder-а: бизнес-код всё ещё пишет SQL вручную, опираясь
  на `dialect` для портируемости.
- Нет миграций.
- Нет привязки типов схемы (`schema/*`) к рантайму драйвера: пока что
  схема — это **только** type-level декларация для будущего слоя
  построения запросов.
- В `RawResult.rows` сидит `Record<string, unknown>` — типизация
  строк по схеме — задача следующего слоя.
