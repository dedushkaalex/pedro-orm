# Codec — Low-Level Design

## Назначение

Codec — это пара функций `encode/decode` для конкретного TS-типа и
конкретного диалекта БД. Codec живёт «над» драйвером: он работает с
**значениями параметров**, в отличие от `Dialect`, который работает с
**SQL-текстом**.

Текущее использование — на уровне определения колонок схемы:
`boolean()`, `timestamp()`, `json<T>()` уже несут с собой соответствующий
codec в поле `_codec`. Применение этих кодеков к параметрам и
результатам — задача будущего query-слоя (на сегодня не реализован);
в `playground/lesson-1.ts` пример вынужденно делает преобразование
вручную (`db.dialect.id === "sqlite" ? 1 : true`).

См. также общую картину в
`docs/hld/01-architecture-overview.md#codec`.

## Публичный API

| Символ           | Сигнатура                        | Назначение                        |
| ---------------- | -------------------------------- | --------------------------------- |
| `Codec<TS, SQL>` | `interface`                      | Контракт кодека                   |
| `passthrough`    | `Codec<unknown, unknown>`        | Без преобразования                |
| `booleanCodec`   | `(d) => Codec<boolean, unknown>` | `boolean` ↔ `0/1` для SQLite      |
| `dateCodec`      | `(d) => Codec<Date, unknown>`    | `Date` ↔ ISO string для SQLite    |
| `jsonCodec<T>`   | `(d) => Codec<T, unknown>`       | `T` ↔ `JSON.stringify` для SQLite |
| `bigintCodec`    | `Codec<bigint, unknown>`         | `bigint` ↔ строка (см. ниже)      |

## Структура

```ts
interface Codec<TS, SQL> {
  readonly encode: (value: TS) => SQL;
  readonly decode: (raw: SQL) => TS;
}
```

- `encode` — TS-значение → значение, которое можно отдать драйверу.
- `decode` — значение из драйвера → TS-значение, ожидаемое бизнес-кодом.

`SQL` параметризован как `unknown` сознательно: он не «строка SQL», а
«какой бы примитив ни ожидал биндинг конкретного драйвера». У PG это
может быть JS-`Date`, у SQLite — строка или число.

## Поведение кодеков по диалектам

| Codec     | PG (`postgres`)                         | SQLite (`sqlite`)                                                                       |
| --------- | --------------------------------------- | --------------------------------------------------------------------------------------- |
| `boolean` | `passthrough` (PG умеет нативный bool)  | `encode: v => v ? 1 : 0`, `decode: raw => raw === 1 \|\| raw === "1" \|\| raw === true` |
| `date`    | `passthrough` (PGlite принимает `Date`) | `encode: v => v.toISOString()`, `decode: raw => new Date(raw as string)`                |
| `json<T>` | `passthrough` (PG поддерживает `jsonb`) | `encode: JSON.stringify`, `decode: JSON.parse`                                          |

Идея общая: **на стороне PG различие нативно, на стороне SQLite
кодируется явно**.

## Нетривиальные решения

### `bigintCodec` — всегда строка, в обе стороны

Не делится по диалекту: и в PG, и в SQLite `bigint` сериализуется в
строку. Причина — JS-`bigint` не входит в JSON, и значимые движки
драйверов по-разному обрабатывают этот тип. Поэтому стратегия одна:
кодер всегда возвращает `v.toString()`, а декодер всегда вызывает
`BigInt(raw)`.

Комментарий в файле говорит: «спорный кейс, лучше явно».

### Codec как функция, а не как значение

Все нетривиальные кодеки — функции `(dialectId) => Codec`. Это нужно,
чтобы привязать конкретное поведение к конкретному движку **на момент
сборки колонки**, а не каждый раз при выполнении запроса. См. поле
`_codec` в `ColumnDef` (`src/schema/columns.ts:14`).

### `passthrough` как общий объект, а не функция

Когда преобразовывать нечего, любая функция-обёртка — лишняя.
`passthrough` — один общий иммутабельный объект, реиспользуется в
ветках всех кодеков.

### Симметрия `encode/decode`

Декодер `boolean` принимает три формы (`1`, `"1"`, `true`), потому что
конкретные адаптеры better-sqlite3 могут отдавать число, строку или
JS-boolean в зависимости от того, как было записано. Один из углов
надёжной двусторонней пары.

## Граничные случаи

- **`null` через codec.** Текущие кодеры не проверяют `null` — это
  ответственность вызывающего: если колонка `nullable`, обработка
  null-значений должна происходить **до** вызова `encode`. В будущем
  query-слое это естественно ляжет в одну функцию-маппер.
- **`JSON.parse` бросает.** Если в SQLite в TEXT-колонку попало
  невалидное JSON-значение, `decode` пробросит исключение — оно будет
  поймано слоем выше и должно стать `CodecError` (тип уже заведён в
  `errors.ts`, см. `docs/lld/errors.md`). На сегодня преобразование в
  `CodecError` не реализовано — это резерв.
- **`bigintCodec` с числовым `raw`.** Если драйвер вернул `number`,
  `BigInt(number)` сработает, но потеряет точность за пределами
  `Number.MAX_SAFE_INTEGER`. Поэтому SQL-слою лучше отдавать `bigint`
  как строку.

## Ограничения

- Codec не интегрирован с `executeRaw` (не вызывается автоматически).
  Сейчас бизнес-код руками передаёт значения, готовые для драйвера.
- Нет рекурсивных кодеков (например, массив JSON).
- Нет валидации после `decode` (например, через `effect/Schema`).
